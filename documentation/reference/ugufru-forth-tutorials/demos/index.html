APPENDIX
Demo Gallery
Fifteen complete programs, and what each one teaches
DEMO GALLERY 1

The tutorial builds up to a guessing game; the demos go the rest of the way. Each is a self-contained Forth program in src/, built with the same fc.py toolchain, and each was written to exercise a particular corner of the CoCo — beam-racing, artifact colour, semigraphics, the two audio paths, the FujiNet clock. Every one ships on build/demos.dsk (see the disk build guide) and runs on real hardware.

The screenshots are captured from XRoar. Click any source & README link to read the code.

Hello World

The minimal kernel smoke-test. Prints BARE NAKED FORTH followed by the kernel version to the VDG text screen with EMIT, then waits for the BREAK key to return cleanly to BASIC.

Exercises: kernel primitives (EMIT, CHAR, RSHIFT), ASCII-to-VDG text output, colon definitions, BEGIN/UNTIL polling, clean exit via lib/bye.fs.

Games & Graphics

Bare Naked Tetris

A full Tetris game in ~555 lines of Forth on a 32×16 SG4 display: a 10-wide playfield plus an info panel with score, lines, next-piece preview, and high scores. Includes collision detection, cascade line-clearing, scoring, difficulty progression, delayed auto-shift key repeat, and hard/soft drop.

Exercises: VDG SG4 colour cells, packed 112-byte rotation table, cascade line-clear, vsync-seeded LCG RNG (lib/rng.fs), number printing (lib/print.fs), compile-time data (lib/datawrite.fs).

Technique: zero-flicker rendering draws the piece at its new position before erasing the old cells (skipping reused cells), so the piece is never absent for even one frame — the headline technique in the codebase.

Bounce — HSYNC Sprite Testbed

Four 7×7 artifact-colour balls bounce around a 128×192 RG6 screen while you switch between three rendering modes — free-running, VSYNC-only, and VSYNC+HSYNC beam-chasing — to see the difference in tearing. A HUD shows the current blanking offset and mode, adjustable with the arrow keys.

Exercises: RG6 artifact-colour pixels (lib/rg-pixel.fs), software sprites (lib/sprite.fs), PIA0 HSYNC beam-chasing, VSYNC sync (lib/screen.fs), 5×7 HUD font.

Technique: draws 2 of 4 balls per frame to stay under the 16.7 ms budget, and waits ~70 HSYNC lines past VSYNC so VRAM writes trail the beam — killing most visible flicker.

Kaleidoscope

A four-way symmetric pattern generator on a 64×32 SG4 pixel grid. Each step picks a random position and plots it mirrored across both axes, using all eight CoCo colours with a 75% plot / 25% erase ratio so patterns build up and slowly recycle.

Exercises: VDG SG4 semigraphics, 16-bit LCG RNG (lib/rng.fs), SAM double-speed toggle, vsync sync (lib/screen.fs).

Technique: computation runs at ~1.78 MHz via the SAM speed bit, dropping back to normal speed before vsync for a clean VDG refresh.

source & README →

Matrix Rain

A Matrix-style digital rain: 32 independent character streams fall at varying speeds down the VDG text screen. Each stream has a bright inverse-video head, a dimmer trail of randomly mutating characters, and a black erasing tail.

Exercises: VDG text mode's three brightness tiers (inverse / normal / SG4-black), four parallel column-state arrays, LCG RNG (lib/rng.fs), SAM double-speed.

Technique: per-frame column updates, mutation, and spawn logic run at SAM double speed, dropping to normal speed before vsync — the same trick as the kaleidoscope.

Text & Interaction

Typewriter

A bare-metal keyboard echo test that types characters to the VDG screen with a visible block cursor and full cursor movement — no BASIC ROM dependency. Handles printable echo, ENTER, CLEAR, backspace, arrow-key movement, and BREAK to exit; a bundled kbdtest.fs also shows each key's hex code.

Exercises: kernel keyboard scan (KEY via direct PIA0 matrix with debounce + SHIFT), EMIT with ASCII-to-VDG encoding and scrolling, AT positioning, direct video-RAM C@/C! for the cursor.

Calculator

An infix integer calculator with an on-screen keypad. A red pixel-font display shows the running value while single-character inverse-video button labels flash on keypress. Supports the four operators, chained operations, clear, and a memory store/recall.

Exercises: VDG SG4 pixel-font digits, direct video-RAM writes, inverse/normal video for the button flash, keyboard input via KEY, 16-bit signed arithmetic, number printing (lib/print.fs).

Display & Colour

VDG Mode Tour

Cycles through all eleven MC6847 display modes (Alpha, SG4, SG6, CG1–6, RG1–6), showing each mode's name plus a representative pattern — colour bars, bands, or checkerboards — rendered with whatever technique that mode supports. Press any key to advance.

Exercises: full VDG/SAM mode switching via a mode table (lib/vdg.fs), four text-rendering strategies (VDG character ROM, RG bitmap font, nibble-expanded CG font, SG6 block-pixel font), compile-time data (lib/datawrite.fs).

Technique: programs SAM V0–V2 (data rate) and F0–F6 (VRAM offset) plus the PIA $FF22 mode pins while preserving bits 0–2, correctly handling SG6's ROM-disable.

source & README →

RG6 Pixel & Sprite Test

A stepped test harness for the RG6 artifact-colour pixel, line, and sprite libraries. Advancing with a keypress, it draws colour bars (shown), a fan of colored lines (white/blue/red by slope), sprite draw/erase, and a rotating trig-driven line sweep, all on a 128×192 RG6 screen under NTSC artifacting.

Exercises: RG6 artifact-colour pixels and rg-line (lib/rg-pixel.fs), software sprites with inline data (lib/sprite.fs), sin table (lib/trig.fs), keypress stepping with BREAK early-exit.

source →

Colored RG6 Text

A diagnostic swatch for the copy-mask colored-text feature. Prints the same line of RG6 artifact text three times, once per colour mask ($FF/$55/$AA), so the mask-to-colour mapping (white/blue/red) can be eyeballed. The $FF row doubles as a white-output regression check.

Exercises: RG6 colored artifact text via the KVAR-RGCOLMASK copy-mask (lib/rg-text.fs, kernel rg-char), font art (lib/font-art.fs), VDG/PIA setup.

Technique: recolouring ANDs each font byte with the colour mask during the glyph copy — no separate colored fonts and zero extra RAM.

Telling Time

Analog + Digital Clock

A future-retro wall clock that fetches the time from a FujiNet RTC, then smoothly sweeps three hands across an oval artifact-colour face drawn from 24 line segments, with quarter ticks and pip markers. A digital YYYY.MM.DD / HH:MM:SS readout sits below, and an "RTC SYNC" flash fires on the once-per-minute re-sync.

Exercises: RG6 line/pixel drawing (lib/rg-pixel.fs), double-buffered beam-trace/draw/restore pipeline (lib/beam.fs), FujiNet RTC (lib/fujinet.fs), sin table (lib/trig.fs), 12 hand-written CODE words.

Technique: five optimisation rounds brought a typical mid-minute frame to 22% of the ~14,917-cycle NTSC field budget; every visible frame fits in one vblank.

FujiNet Time

Fetches a 6-byte date/time block from the FujiNet device over the cartridge's bit-banger DriveWire driver, then formats and prints it as YYYY.MM.DD HH:MM:SS on the VDG text screen. A focused test of the FujiNet time path.

Exercises: FujiNet fn-time over DriveWire bit-banger (lib/fujinet.fs), an inline CODE word with an RMB data buffer, VDG text output, 2-digit/4-digit decimal formatting.

Note: requires a real (or emulated) FujiNet as the time source — under plain XRoar it waits on a device that never answers.

Sound

DAC Sound

An interactive tone-and-effect showcase for the 6-bit DAC at $FF20 — the audio path BASIC's SOUND uses, driven directly from Forth CODE words. Keys trigger square/saw/triangle/sine tones and named effects (beep, zap, boom, chirp, dock, hit) via a KEY dispatch.

Exercises: 6-bit DAC output ($FF20), explicit PIA0/PIA1 setup, sound primitives snd-tone/snd-saw/snd-sin/snd-noise (lib/sound.fs), sin table (lib/trig.fs), KEY dispatch.

Technique: because snd-tone does its own PIA init, it runs identically in ROM-mode and all-RAM kernels; the README documents measured cycle costs (1 ms ≈ 890 cycles) for every effect.

source & README →

1-Bit Sound

Exercises the CoCo's second audio path — the single-bit output at $FF22 bit 1, mixed into the same bus as the DAC. Keys play square tones, an ascending sweep, a click train, and a noise burst, plus an A/B key that plays the same tone through the DAC then the 1-bit path for direct comparison.

Exercises: 1-bit sound via read-modify-write toggling of $FF22 bit 1 (snd-tone1/snd-click1/snd-noise1), PIA1 DDR setup, RNG-gated noise, shared DAC library (lib/sound.fs).

Technique: RMW toggling is mandatory because $FF22 also carries VDG mode bits and RS-232 IN — a plain store would scramble the screen mode; the 1-bit path is fixed 50% duty, so square tones and clicks only.

source & README →

Async Sound Engine

A showcase of the non-blocking sound engine: every effect is produced cooperatively one sample at a time, so an on-screen marker keeps animating and the keyboard stays live while sound plays. Keys trigger tones, sweeps, noise, ring-modulated bell/gong timbres, and queued multi-note effects.

Exercises: cooperative HSYNC-polled DAC output (lib/async-sound.fs), phase-accumulator wavetable voice (lib/wavetable.fs), per-frame pitch slide / amplitude envelope / ring modulation, LFSR white noise, a small note queue for sequencing.

Technique: the engine polls the HSYNC flag ($FF01) to clock the DAC instead of taking an interrupt or busy-waiting — no kernel changes and no blocking, so gameplay continues during playback (~1.1 KB plus one runtime-generated wavetable).
