THE XROAR EMULATOR 1

Every program in this book was run in XRoar long before it touched real hardware. XRoar is a cycle-accurate emulator for the Dragon, the Tandy Color Computer (1/2/3), and the MC-10, written by Ciaran Anscomb. For our purposes the important word is cycle-accurate: the CoCo 2 it emulates behaves closely enough to real iron — same 0.89 MHz 6809, same SAM and VDG timing — that a program which looks right in XRoar looks right on the desk. That is what makes the tight edit–compile–run loop possible.

We drive it entirely from the command line. There is a GUI, but the CLI is what fits the workflow: a single make run pipes a freshly compiled binary straight into the emulator, and the same flags go into scripts, Makefiles, and the screenshots in this book.

XRoar isn't part of Bare Naked Forth — it's a separate, excellent project. Install it from 6809.org.uk/xroar (or Homebrew), and keep your CoCo ROM images in ~/.xroar/roms/.

How We Run It

The canonical invocation — the one behind make run — loads the two BASIC ROMs and executes a binary: 

$ xroar -machine coco2bus -ram 32 \
        -bas ~/.xroar/roms/bas12.rom \
        -extbas ~/.xroar/roms/extbas11.rom \
        -run build/combined.bin

Piece by piece:

  • -machine coco2bus — a CoCo 2 (NTSC, 6809). This is our reference machine.
  • -ram 32 — 32 KB, matching the default ROM-mode build. All-RAM demos use -ram 64.
  • -bas / -extbas — the Color BASIC and Extended Color BASIC ROM images. In ROM mode our kernel lives alongside a live BASIC, so both are loaded.
  • -run <file> — load a DECB binary and execute it immediately, exactly as LOADM"…":EXEC would.

The demo Makefiles wrap this in a run target with two knobs: XROAR_RAM (32 or 64) and XROAR_EXTRA for per-demo flags — most commonly -kbd-translate, which maps your host keyboard onto the CoCo matrix so interactive demos (the typewriter, the calculator) read the keys you expect.

Loading a Disk vs. Running a Binary

-run is the fast path, but it is not the whole story. To exercise a real Disk BASIC image — the build/demos.dsk from the disk-build guide — mount the disk and boot into Disk BASIC instead: 

$ xroar -machine coco2bus \
        -bas ~/.xroar/roms/bas12.rom \
        -extbas ~/.xroar/roms/extbas11.rom \
        -dos ~/.xroar/roms/disk11.rom \
        build/demos.dsk
# then at the OK prompt:  LOADM"TETRIS":EXEC
A test path that hides bugs: -run file.bin loads the binary directly and bypasses DECB entirely. That's great for speed, but it means -run will happily execute a program that would hang when loaded from a real disk with LOADM. Before claiming a size- or layout-affecting change is safe, build the .dsk and load it the real way. Know which of your harnesses give false confidence.

Cartridges

Some demos want a cartridge ROM in the slot. The clock and FujiNet-time demos read a real-time clock that arrives over an HDB-DOS / DriveWire cart, so their Makefiles add: 

XROAR_EXTRA = -cart-rom ~/.xroar/roms/hdbdw3cc2.rom -cart-autorun -kbd-translate

Debugging from the Command Line

XRoar's command-line build carries real debugging instruments. Three are worth knowing.

The GDB Stub

Launch with -gdb and XRoar opens a GDB server on 127.0.0.1:65520. Point a host GDB at it and you can step the 6809, read and write memory, and set breakpoints and watchpoints on the running program: 

$ xroar -machine coco2bus -gdb … -run build/combined.bin &
$ gdb
(gdb) target remote :65520

The stub speaks the full 6809 register set (CC, A, B, DP, X, Y, U, S, PC — plus the HD6309 extras on a 6309 machine) and has XRoar-specific extensions to read and write the SAM register — invaluable when a demo toggles SAM speed or VRAM-offset bits and you need to see the current state.

Instruction Trace

A build with tracing enabled accepts -trace, streaming each instruction as it executes. It's verbose — pipe it to a file and grep — but when a threaded Forth word is going somewhere it shouldn't, watching the actual NEXT/DOCOL dispatch is the fastest way to see where the thread derails.

Snapshots

XRoar can save and reload machine snapshots, so a hard-to-reach state (deep into a game, a specific corrupted screen) can be frozen and returned to instead of reproduced by hand each time.

Working Habits

A few disciplines keep the emulator a help rather than a distraction — the same ones the rest of this project runs on.

One launch, then observe. Always pkill -9 xroar before relaunching, and never spam-launch: start one instance, look at the result, then iterate. Firing the emulator in a loop just buries the frame you needed to see.

Screenshots, not synthetic keystrokes. On macOS, injected keystrokes don't reliably reach XRoar's emulated keyboard — so we verify by looking. A screen-capture tool grabs the XRoar window after each run, which is how every screenshot in the demo gallery was made and how visual bugs — flicker, a wrong colour, an off-by-one sprite — get caught. When a demo needs a key pressed to advance, that's a moment to drive it by hand and watch.

XRoar also builds to WebAssembly (--enable-wasm) — the same emulator, in a browser. That is the door to running these demos inline in the docs one day, without leaving the page.
