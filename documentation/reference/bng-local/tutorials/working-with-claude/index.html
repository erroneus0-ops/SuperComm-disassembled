APPENDIX
Working with Claude
The method behind every line of this project
WORKING WITH CLAUDE 1

One fact shaped everything else: every line of this project was written by Claude. The kernel, the compiler, the demos, this page. The human never typed a line of code. Around 97% of the commits name Claude as co-author; the rest are trivial.

But typing isn't coding. Claude does the typing, and often the first cut of the ideas and the words too. The human does the rest: deciding what to build, weighing each change, keeping what works, sending back what doesn't. A lot is kept as-is. That's real coding, just without the keystrokes.

Both sides contribute. But the human has the final say. They're the gateway. Nothing ships without their approval.

This page is how that partnership works. None of it is a nice-to-have. These are the moves that put a real toolchain on a 1982 machine. Skip one and the model drifts. The examples are from the CoCo, but the method fits almost any project you'd build with Claude Code.

You don't need any of this just to use Bare Naked Forth. It's how we built it. Take what's useful.

Two Roles

Think of it as two roles.

You're the architect. You see the big picture. You decide when to refactor and when to ship, and you catch the thing the model can't see from inside the code.

Claude is the tireless junior. Fast, never tired, knows the whole manual at once. And it never forgets between sessions, because the memory lives in the repo, not in its head.

Your job shifts from typing to deciding and judging. That's still coding, and it's still work: the hard part was always knowing what to build and spotting what's wrong.

The Loop

The cycle is fast, and each pass takes seconds:

  1. Write Forth in a .fs file.
  2. Compile it with fc.py into a .bin.
  3. Run it in XRoar, a spot-on CoCo emulator (see the XRoar appendix).
  4. Ship it to real hardware when it's right: same file, no changes.

A full rebuild takes under a second. You change something, run it, and see it before your next thought. That speed is what makes the whole thing work. 

> Can you build and run this project?
WORKING WITH CLAUDE 2

The Repo Is the Memory

Claude starts each session with no memory of the last one. The repo is the only thing that carries over, so it has to hold anything worth keeping. The rule: if it matters, write it down in the repo, in plain words. Whatever stays in the chat is gone when the session ends.

Each kind of knowledge has a home. Conventions and how-to live in CLAUDE.md, the work to be done lives in issues.jsonl, and how the code works lives in the docs. That's the first place Claude should look, ahead of the web.

Have Claude keep the docs up to date. What it learns now is then ready for whoever picks it up next. 

> Let's create a git repo with a README.md file.

The Repo Is Your Undo Button

Every working version is a commit, a save point you can always return to. So you're never stuck with what Claude just did. Don't like where the last hour went? Roll back to the last good commit and try again, as many times as it takes.

It's a safety net, too. An API call drops, a battery dies, a run goes off the rails. No disaster. You lose one session, not the project. Commit often. The repo is the one thing you can't lose.

Break Up Plans into Issues

Work starts with a conversation. Often a long one: a real back-and-forth where each side deepens the other's understanding of what to build and why. It usually ends with Claude writing a plan, and the urge is to let it start building. Don't. Turn the plan into issues first.

Instead of executing Claude's plan, ask it to break up the plan into trackable issues: one per change, written to issues.jsonl, each holding enough detail to complete later. That last part is the point: it moves the plan into the repo, ready to start anytime we choose. A later session with no memory of this conversation can still pick it up. Claude does this well when you ask.

Why write the issue first? It pins the idea while it's fresh, gives Claude a clear target, and makes "done" mean something. Writing issues after the work loses all of that.

Filing an issue doesn't commit you to building it now. Sometimes you create, build, and close one in a single session; other times it waits. That flexibility is the whole trick for big jobs: break them into pieces you can pick up whenever. Once the work is a list, you can reorder it, cut it, and spot what runs in parallel.

Last, have Claude boil the plan down to a short roadmap (in the README or a ROADMAP.md) listing what's next, in order. That roadmap plus the issues is all the next session needs to start. 

> Hi Claude, what's next on the roadmap?
WORKING WITH CLAUDE 3

Test Before You Trust

Compiling isn't done. Done is watching it run. Claude drives XRoar itself (kill, rebuild, run) and grabs a screenshot to check the result. You step in for the feel: is the motion smooth, does the tone sound right, does it hold up on a real CoCo? The loop stops and asks instead of guessing.

Know When to Stop

Let Claude run on its own through the busywork: edits, compiles, greps. Have it stop for the real calls: a design choice, a look-and-see, anything hard to undo.

Build Your Own Tools

Claude writes code cheaply, so flip the usual default: need a tool? Build it instead of adding another external dependency.

Every outside dependency is code you don't control. Here, almost everything is in the repo, readable and yours: the whole toolchain is a ~500-line kernel and a ~400-line Python compiler.

Make Every Byte Count

The 64 K limit isn't a problem. It's the point. It makes every choice deliberate. Some code goes in the kernel, some in the app, and that's a real trade: a CODE word sits inline in the app; a KCODE word sits in spare kernel space, keeping the app small. Moving eight hot words to KCODE once freed 397 bytes in the app, and grew the kernel by the same amount. Name the trade out loud and both sides stay honest.

What Only You Can Do

A few of your moves are worth more than all the typing they replace. Three keep coming up:

  • Point at the truth. Without specific answers, Claude guesses.
  • Say when to push and when to pivot. Stuck on a hard bug, Claude may offer to fall back to what already worked. "No, keep trying, the solution is not that complicated. Do you need help?" is your call to make, and it's often the difference between the result you wanted and one that merely runs.
  • See what it can't. Claude can't see its own output. A one-pixel glitch every fourth frame, a sprite that sits wrong: those live on the screen, not in the code, and only your eye catches them. Flag it, and Claude can chase the math.

None of this is the tool falling short. It's the shape of the team. Claude brings tireless, exact hands; you bring the pointer, the call, and the eye.

Why It Works Anywhere

The CoCo made a perfect test bed: small enough to hold in your head, fast to build, honest to emulate, instant to see. Its limits, once a burden, are what make it easy to work on. A vague idea has nowhere to hide.

But the method isn't about the CoCo. A repo that's both memory and undo button; the habit of checking, measuring, and testing; a human who decides and sees, plus a model that never tires. All of that travels to any project. The CoCo just makes it easy to see. Use it, and any project becomes one where the plan is written down, the choices are on purpose, and nothing ships that you didn't want. That's how this one got here, and why the next will too.
