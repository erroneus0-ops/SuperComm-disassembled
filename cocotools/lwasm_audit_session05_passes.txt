FINDING 3
--------------------------------------------------------------------------------
FILE:      cocotools/passes.py  (_do_pass4_aux)
C SOURCE:  lwasm/pass4.c  (do_pass4_aux)
ISSUE:     Python adds "if sl is None: break" when the cursor runs off the
           end of the line list while searching for the next unresolved
           instruction. C has no equivalent check at this point — if this
           path were ever reached in C, it would dereference sl->addr on a
           NULL pointer (a crash).
SEVERITY:  Significant (uncertain reachability, but the failure MODE
           differs meaningfully if it is reachable)
EVIDENCE:  C:
             for ( ; sl && sl -> len != -1; sl = sl -> next) { ... }
             // no NULL check here before:
             as -> cl = sl;
             lwasm_reduce_expr(as, sl -> addr);   // crashes if sl==NULL
           Python:
             while sl and sl.len != -1: ...
             if sl is None:
                 break                              # <- returns quietly
             ...
ANALYSIS:  Under correct invariants (cnt accurately reflects the number of
           still-unresolved lines reachable by continuing to scan forward
           from sl's current position), this path should be unreachable —
           if cnt > 0, there should always be at least one more line with
           len == -1 still ahead of sl. I have NOT proven this invariant
           holds in all cases (it depends on pass3's behavior and on
           nothing upstream ever double-counting or mis-counting cnt).
           IF the invariant ever breaks (due to some other bug, including
           possibly one not yet found in this audit), the two
           implementations fail differently in a way that matters:
             - Real lwasm: crashes immediately. Loud, obvious, gets
               investigated right away.
             - cocotools: silently breaks out of the loop and returns.
               do_pass4 (force=1) returns normally with cnt still > 0 —
               meaning there ARE unresolved instructions — but no
               E_INSTRUCTION_FAILED error was ever raised for them, and
               nothing downstream in this function reports the problem.
           This is the same "defensive guard changes the failure mode from
           loud-crash to silent-continue" pattern noted in Session 3
           (Finding 2) and Finding 2 above. On its own any single instance
           looks like reasonable defensive coding; as a repeated pattern
           across the codebase it means: if there IS an invariant-breaking
           bug anywhere upstream, cocotools is structurally more likely to
           mask it (silent wrong output) rather than surface it (crash),
           compared to real lwasm. Worth deciding as a deliberate policy
           (e.g., these guards should assert-and-report rather than
           silently break) rather than leaving each one as an ad hoc
           choice.

FINDING 4 — HIGH VALUE
--------------------------------------------------------------------------------
FILE:      cocotools/passes.py  (do_pass5)
C SOURCE:  lwasm/pass5.c  (do_pass5)
ISSUE:     The real lwasm C source itself appears to contain a genuine bug
           in this function — and the Python translation faithfully
           reproduces HALF of that bug while silently correcting the other
           half.
SEVERITY:  Significant (this is about faithfulness-of-translation policy,
           not really about "is the Python correct" — see analysis)
EVIDENCE:  C (pass5.c, the address-resolution recovery loop):
             for (cl = sl; cl; cl = cl -> next)
             {
                 as -> cl = sl;
                 lwasm_reduce_expr(as, sl -> addr);      // uses sl, not cl
                 if (exprok(as, cl -> addr))              // checks cl->addr
                 {
                     if (0 == --cnt) return;
                 }
                 lwasm_reduce_expr(as, sl -> daddr);     // uses sl, not cl
                 if (exprok(as, cl -> addr))              // BUG: checks
                 {                                        // cl->addr AGAIN,
                     if (0 == --cnt) return;               // not cl->daddr
                 }
             }
           This loop advances cl but always reduces/checks against the
           fixed sl (not the advancing cl) for BOTH sub-checks, and the
           second sub-check tests cl->addr a second time instead of
           cl->daddr. This looks like a copy-paste error in the original
           lwtools source (checked visually multiple times against the
           actual fetched file to rule out a transcription mistake on my
           end — the text above is exactly what's in pass5.c).

           Python (passes.py, do_pass5):
             cl = sl
             while cl:
                 as_.cl = sl
                 as_.reduce_expr(sl.addr)                 # matches C: uses sl
                 if _exprok_p5(as_, cl.addr):              # matches C: cl.addr
                     cnt -= 1
                     if cnt == 0: return
                 as_.reduce_expr(sl.daddr)                 # matches C: uses sl
                 if _exprok_p5(as_, cl.daddr):              # DIFFERS FROM C:
                     cnt -= 1                               # checks cl.daddr,
                     if cnt == 0: return                     # not cl.addr again
                 cl = cl.next

ANALYSIS:  Python faithfully reproduces the "reduces sl instead of the
           advancing cl" half of the bug, but silently corrects the
           "checks addr twice instead of addr-then-daddr" half. This is
           very likely an artifact of how the translation was done —
           checking cl->addr twice in a row reads so obviously like a typo
           that a human or AI translator naturally "fixes" it without
           registering that as a deliberate policy decision, while the
           sl-vs-cl substitution is less visually obvious as a mistake and
           got carried over unchanged.
           PRACTICAL CONSEQUENCE: for any source program that actually
           exercises this recovery loop (i.e., has address expressions
           that don't resolve during pass3/4 and need pass5's fallback
           walk), the set of lines whose daddr gets checked/counted as
           resolved will differ between cocotools and real lwasm, because
           real lwasm never actually re-checks daddr here at all — it
           checks addr twice and never notices if daddr specifically was
           the unresolved piece. This could mean:
             - cocotools correctly resolves and clears more lines per
               iteration than real lwasm would (since it's actually
               checking the right field), potentially finishing in fewer
               iterations, and/or
             - cocotools raises E_LINED_ADDRESS in different cases (or
               fails to) compared to real lwasm, since real lwasm's
               daddr-resolution tracking through this specific loop is
               effectively broken and relies on other passes to have
               already sorted daddr out.
           This is squarely the kind of divergence the project's own
           "faithful translation, bugs included" principle is meant to
           catch — not because the Python is behaving badly (arguably it's
           behaving BETTER), but because "better" here is still
           "different from what real lwasm actually does," which
           undermines exact round-trip verification against real lwasm
           output for any test case that reaches this path.
DECISION NEEDED (not mine to make): does the project want byte-exact
           behavioral parity with real lwasm including this bug, or is
           silently improving on a discovered upstream bug acceptable?
           Either is defensible — but right now it happened by accident
           rather than by decision, which is the actual issue.
