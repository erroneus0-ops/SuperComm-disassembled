FINDING 2
--------------------------------------------------------------------------------
FILE:      cocotools/listing.py
C SOURCE:  lwasm/list.c, lwasm/lwasm.h
ISSUE:     PRAGMA_NOLISTCODE, checked first thing in the per-line listing
           loop (skip the line entirely from the listing if set), does not
           exist anywhere in real lwasm — confirmed via grep across both
           list.c and lwasm.h. It's a cocotools-only addition (bit 29 in
           the pragma bitfield, consistent with the established pattern of
           appending inventions after the real bits rather than inserting
           them and shifting numbering — same safe pattern seen with the
           PHASE/DEPHASE error codes in Session 1).
SEVERITY:  Minor (does not affect assembled binary output; only affects
           .lst file structure, and only when a cocotools-specific pragma
           name is deliberately used — no real lwasm source would ever
           reference it)
EVIDENCE:  grep for "NOLISTCODE" across list.c and lwasm.h: no matches.
           cocotools/lwasm_types.py: PRAGMA_NOLISTCODE = 1 << 29.
           cocotools/listing.py: checked as the very first condition in
           _do_list_inner's per-line loop.
CONSEQUENCE: purely additive — using this pragma in cocotools-targeted
           source gives a convenience not available in real lwasm. Worth
           documenting in DESIGN.md as an intentional extension (same
           recommendation as for PHASE/DEPHASE) so it's clearly understood
           as "cocotools-only" rather than discovered by surprise later.
