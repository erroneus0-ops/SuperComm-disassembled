FINDING 2
--------------------------------------------------------------------------------
FILE:      cocotools/input_system.py  (open)
C SOURCE:  lwasm/input.c  (input_open, prefix-determination section)
ISSUE:     When an input specifier contains a colon but the text before it
           is neither "include" nor "file" (or the colon is the very first
           character, i.e. an empty prefix), real lwasm raises an explicit
           error. Python silently falls through to treating the ENTIRE
           original string — prefix, colon, and all — as a literal
           filename.
SEVERITY:  Minor (narrow edge case — malformed or unrecognized prefixes
           are unlikely to occur in normal use — but a real, demonstrable
           behavioral difference)
EVIDENCE:  C:
             for (s2 = s; *s2 && (*s2 != ':'); s2++) ;
             if (input_isabsolute(s) || !*s2)
                 t -> type = input_type_file;
             else {
                 ts = lw_strndup(s, s2 - s);
                 if (!strcmp(ts, "include")) type = input_type_include;
                 else if (!strcmp(ts, "file")) type = input_type_file;
                 else type = input_type_error;   // <- explicit error type
             }
           input_type_error is not handled by either switch case in
           input_open, so execution falls through to:
             lw_error("Cannot figure out how to open '%s'.\n", ...);
           — a hard, explicit, clearly-worded error.

           Python:
             if not _is_absolute(s):
                 colon = s.find(':')
                 if colon > 0:              # <- note: strictly > 0
                     prefix = s[:colon]
                     if prefix == 'include': ...
                     elif prefix == 'file': ...
                     # no else branch — input_type stays at its default
                     # (INPUT_TYPE_FILE) and path stays as the full
                     # original string, unprefixed-and-uncolon-stripped
           Two related divergences from this:
             (a) colon == 0 (string starts with ':') — C computes an
                 empty-string prefix and correctly falls to
                 input_type_error; Python's "colon > 0" check treats
                 colon==0 the same as "no colon found," silently defaulting
                 to file type with the leading colon kept as part of the
                 filename.
             (b) colon > 0 but prefix text doesn't match "include" or
                 "file" — C: input_type_error → hard error. Python: silent
                 fallthrough to file type using the whole original string
                 (prefix + colon included) as the filename, which will
                 most likely just fail to open with a generic OS-level
                 "file not found" error rather than lwasm's specific,
                 clearer diagnostic.
CONSEQUENCE: mostly a difference in ERROR QUALITY/CLARITY rather than
           silently wrong output (a genuinely malformed input spec will
           still fail to assemble either way, just with a less specific
           error message from cocotools). Flagging for completeness per
           the "let's be complete" directive, but this is lower priority
           than findings that produce silently different correct-looking
           output.

FINDING 3
--------------------------------------------------------------------------------
FILE:      cocotools/input_system.py  (open, INPUT_TYPE_INCLUDE branch)
C SOURCE:  lwasm/input.c  (input_open, input_type_include case)
ISSUE:     Real lwasm has a distinct, explicit code path for an ABSOLUTE
           path given after an "include:" prefix (e.g.
           "include:/full/path/to/file.asm") — it opens that path directly
           and, if it fails, checks a specific "ignore this error"
           condition (IGNOREERROR, defined as "file not found AND we're in
           dependency-generation/preprocess mode") before deciding whether
           to hard-error or just set a soft as->fileerr flag and continue.
           Python has no equivalent explicit branch, no equivalent
           IGNOREERROR condition anywhere in this file, and no code path
           that ever sets self.fileerr (it's initialized in AsmState but
           never assigned afterward, confirmed by search).
SEVERITY:  Significant (specifically for anyone using or wanting to use a
           dependency-listing / "don't fail on missing includes" mode —
           otherwise low impact, since path RESOLUTION for the common case
           happens to still work, see below)
EVIDENCE:  C (lwasm.h): #define IGNOREERROR (errno == ENOENT &&
           (as->preprocess || as->flags & FLAG_DEPENDNOERR))
           — this is specifically lwasm's "-M"-style dependency-generation
           support: when just listing what files a source depends on,
           missing include files shouldn't be fatal.
           cocotools DOES have the corresponding state fields
           (AsmState.preprocess, FLAG_DEPENDNOERR constant in
           lwasm_types.py) but they are not wired into any error-handling
           logic in input_system.py — confirmed via search, no
           IGNOREERROR-equivalent condition or self.fileerr assignment
           exists anywhere in the file.
ACCIDENTAL PARTIAL BEHAVIOR MATCH: Python's open() doesn't special-case
           absolute paths within the include branch at all — it just does
           os.path.join(cur_dir, path) and, if that fails,
           os.path.join(inc_dir, path) for each entry in include_list.
           Because Python's os.path.join silently discards all earlier
           components whenever a later component is itself an absolute
           path, os.path.join(anything, "/abs/path.asm") always evaluates
           to exactly "/abs/path.asm" — so path RESOLUTION for an absolute
           include target happens to land on the same file C would open,
           purely as a side effect of os.path.join's documented behavior,
           not because Python deliberately replicated C's explicit
           absolute-path branch.
           What does NOT match: if that absolute path fails to open,
           Python retries the exact same (unchanged) absolute path once
           per entry in include_list (a harmless but pointless retry
           loop), then raises a hard IOError unconditionally — with no
           equivalent of C's IGNOREERROR graceful-continuation option.
CONSEQUENCE: for ordinary assembly (no dependency-generation mode in use),
           this likely doesn't matter — resolution of a working absolute
           include path succeeds the same way either side. It only
           surfaces if (a) a dependency-generation/preprocess-only mode is
           ever wanted for cocotools (currently there's no evidence this
           mode is wired up anywhere, so may be moot), or (b) someone
           relies on assembly continuing gracefully past a missing
           absolute include in that mode specifically.
