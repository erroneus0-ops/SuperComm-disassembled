FINDING 2
--------------------------------------------------------------------------------
FILE:      cocotools/lwasm_core.py  (register_symbol)
C SOURCE:  lwasm/symbol.c  (register_symbol)
ISSUE:     One line diverges from C, in a direction that makes Python safer
           than C rather than less faithful — flagging per the audit's
           "note even benign-seeming deviations" instruction rather than
           because it's clearly wrong.
SEVERITY:  Minor
EVIDENCE:  C:      if (islocal) context = cl -> context;
                   /* no NULL check on cl here */
           Python: if islocal: context = cl.context if cl else -1
           If register_symbol is ever called with cl == NULL while
           registering a symbol whose name contains '@', '?', or an
           unescaped '$' (i.e., islocal becomes true with no line context),
           real lwasm would dereference a NULL pointer at this line — a
           crash. Python instead falls back to context = -1 and continues.
           I have NOT yet confirmed whether this code path is actually
           reachable in practice (i.e., whether any real call site invokes
           register_symbol with cl=NULL for a local-looking symbol name) —
           noting it as an open question rather than asserting it matters.
           If it IS unreachable in practice, this is a harmless defensive
           addition and can be disregarded. If it IS reachable, it's a case
           where cocotools would silently do something different (proceed
           with context=-1) rather than replicate the C crash — worth a
           deliberate decision either way rather than an accidental one.
NEXT STEP: grep all call sites of register_symbol in pass1.c / pass2.c /
           pseudo.c (the C source) for any that could plausibly pass a
           NULL line pointer, to settle whether this is reachable.
