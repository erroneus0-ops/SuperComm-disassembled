FINDING 1
--------------------------------------------------------------------------------
FILE:      cocotools/pseudo.py
C SOURCE:  lwasm/pseudo.c
ISSUE:     INCLUDEBIN — a real, functioning lwasm pseudo-op that includes
           the contents of a raw binary file at assembly time — is present
           in the Python instruction table (instab.py) and dispatches
           successfully, but its parse/resolve/emit functions are all
           no-op stubs. Custom error codes were added specifically for its
           error cases (E_INCLUDEBIN_ILL_START, E_INCLUDEBIN_ILL_LENGTH)
           but are never actually raised, since the parse stub never
           reaches the code that would check start/length validity.
SEVERITY:  Critical (in terms of practical impact — this is a commonly
           used directive, e.g. for embedding graphics/sound data — even
           though it's openly marked incomplete rather than silently
           wrong)
EVIDENCE:  C: pseudo.c defines real pseudo_parse_includebin (line 1464) and
           pseudo_emit_includebin (line 1513) with actual file-reading
           logic and start/length validation.
           Python:
             def pseudo_parse_includebin(as_, cl, operand):
                 return _stub_parse(as_, cl, operand)
             def pseudo_resolve_includebin(as_, cl, force):  pass
             def pseudo_emit_includebin(as_, cl):            pass
           where _stub_parse is:
             def _stub_parse(as_, cl, operand):
                 cl.len = 0; return ''
           CONSEQUENCE: source using INCLUDEBIN "file.bin" would assemble
           without any error message, but the included binary data would
           simply be absent from the output (cl.len = 0 means the
           assembler thinks this line contributes zero bytes). This is
           worse than an error in one specific sense: it's silent from the
           tool's point of view, even though the code itself is honestly
           labeled — the risk is entirely on the human not noticing the
           output is short the expected bytes.

FINDING 2
--------------------------------------------------------------------------------
FILE:      cocotools/pseudo.py (and cocotools/pass1.py, which references a
           cocotools/macro.py module that does not exist in the repo)
C SOURCE:  lwasm/pseudo.c
ISSUE:     Full inventory of pseudo-ops that are present in the instruction
           table and dispatch successfully, but whose parse/resolve/emit
           functions are all no-op stubs:
             MACRO / ENDM           (macro DEFINITION)
             SECTION / ENDSECT
             STRUCT / ENDS
             EXTERN / EXPORT / EXTDEP
             PRAGMA / *PRAGMA / *PRAGMAPUSH / *PRAGMAPOP
             REORG
             OS9
             MOD / EMOD             (OS-9 module boundaries)
             INCLUDEBIN / INCLUDESTR (see Finding 1)
             SETSTR
             IFPRAGMA / IFSTR
SEVERITY:  Significant (transparently documented, but broader in scope
           than DESIGN.md/CLAUDE_MANIFESTO.md currently record)
EVIDENCE:  Each function above is defined exactly once in pseudo.py, as:
             def pseudo_parse_XXX(as_, cl, operand): return _stub_parse(...)
           with no later override anywhere in the file (confirmed via
           definition-count check). Additionally, pass1.py's own header
           comment states macro EXPANSION (not just definition) is
           "deferred to macro.py" via as_.expand_macro(cl, sym) and
           as_.add_macro_line(line) — but no macro.py file exists in the
           repo (confirmed 404 on cocotools/macro.py). So macro support is
           incomplete on both the definition side (MACRO/ENDM stubbed) and
           the invocation side (the module that would expand a macro call
           doesn't exist yet).
IMPORTANT CROSS-REFERENCE TO SESSION 3: Session 3 verified the pragma-
           checking logic (curpragma(), the PRAGMA_* bit tests throughout
           the codebase) is faithfully translated and correct. This
           session's finding means the PRAGMA pseudo-op itself — the thing
           that would actually let assembly source SET those pragma bits —
           doesn't yet do anything. So the correctness verified in Session
           3 currently has limited practical reach: pragmas can only be at
           whatever their default/initial state is, since there's no
           working directive to change them from source. (I have not
           checked whether pragmas can be set via a command-line-flag
           equivalent in cocotools, which would be a separate path not
           going through this stub — flagging as an open question.)

FINDING 3 (POSITIVE — VERIFIED CORRECT, TIES TO SESSION 2)
--------------------------------------------------------------------------------
FILE:      cocotools/pseudo.py  (pseudo_parse_ifeq and siblings)
C SOURCE:  lwasm/pseudo.c  (pseudo_parse_ifeq and siblings)
ISSUE:     None found. Confirms and closes out a question raised in
           Session 2.
SEVERITY:  N/A
EVIDENCE:  pseudo_parse_ifeq:
             e = as_.parse_cond(p)
             if e: as_.reduce_expr(e)
             if e and e.intval() != 0:
                 as_.skipcond = 1; as_.skipcount = 1
           This parses a single plain arithmetic expression (via the
           normal lw_expr grammar) and does the equality test as a
           separate post-hoc step, exactly matching the real C design
           (lwasm_parse_cond + a comparison done in the pseudo-op handler,
           NOT via a comparison operator inside the expression grammar).
           This confirms Session 2 Finding 2's six invented comparison
           operators (==, !=, <, <=, >, >=) in lw_expr.py's grammar are not
           used anywhere by the IF-family conditionals — they appear to be
           a pure, unconsumed addition with no internal caller found so
           far, reinforcing that they should likely just be removed rather
           than needing any consumer to be preserved.
